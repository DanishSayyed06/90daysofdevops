### Task 2: CI vs CD
Research and write short definitions (2-3 lines each):
1. **Continuous Integration** — what happens, how often, what it catches

- Continuous Integration is the practice where developers frequently merge their code changes into a central repository, usually multiple times a day. Each merge triggers an automated build and test process, which catches bugs, integration errors, and code conflicts early before they become larger problems.

`Example` ---> A team of web developers working on a shopping app pushes their code to a shared repository every few hours. An automated system immediately runs a suite of tests to ensure the new "checkout" button doesn't break the existing "login" page.


2. **Continuous Delivery** — how it's different from CI, what "delivery" means

- Continuous Delivery picks up where CI leaves off by automatically preparing code changes for a release to production at any time. The key difference is that while the code is always in a "deployable" state, the actual release to customers requires a final manual trigger or human approval.

`Example`---> A mobile app company runs automated tests on every update. Once the tests pass, the new version is packaged and ready to go in the app store, but the product manager clicks a "Publish" button only after reviewing the release notes and timing.


3. **Continuous Deployment** — how it differs from Delivery, when teams use it


- Continuous Deployment is the most automated version of the cycle, where every change that passes the automated pipeline is immediately released to the production environment without any human intervention. Teams use this when they have very high trust in their automated testing and want to get features into users' hands as fast as possible.

`Example` ---> A streaming service like Netflix uses this approach. When an engineer fixes a minor bug in the video player, the code is automatically tested and pushed live to millions of users globally within minutes, with no one having to manually approve the release.

### Task 3: Pipeline Anatomy
A pipeline has these parts — write what each one does:
- **Trigger** — what starts the pipeline
- **Stage** — a logical phase (build, test, deploy)
- **Job** — a unit of work inside a stage
- **Step** — a single command or action inside a job
- **Runner** — the machine that executes the job
- **Artifact** — output produced by a job

`Trigger` ---> The specific event or condition (like a code commit or timer) that automatically initiates the pipeline execution.

`Stage` ---> A high-level organizational container that groups related tasks into a sequential phase of the process.

`Job` ---> An individual set of instructions that runs within a stage, typically executed on a single machine.

`Step` ---> A discrete, granular command or script execution within a job that performs a specific, atomic task.

`Runner` ---> The computing environment (server, container, or VM) that carries out the actual instructions defined in a job.

`Artifact` ---> The persistent file, binary, or report generated by a job that can be passed to subsequent stages or stored for later use.
